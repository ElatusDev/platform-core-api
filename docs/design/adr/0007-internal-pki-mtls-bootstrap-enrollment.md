# ADR-0007: Internal PKI with mTLS and Bootstrap Token Enrollment

**Status**: Superseded
**Date**: 2025-02-21
**Deciders**: ElatusDev

## Context

AkademiaPlus is a multi-tenant educational platform handling student PII, financial
transactions (billing/POS), and sensitive operational data across 15+ modules. The platform
is evolving toward a multi-service architecture with the `certificate-authority` module as
a dedicated internal service alongside `platform-core-api`, with future services including
notification, ETL, and audit systems.

Current state:
- TLS certificates are manually generated via `keytool` commands (documented in `certs_makani.rtf`)
- Private key passwords stored in plaintext `.env` file
- CA service block in `docker-compose.dev.yml` uses inline `openssl` commands on `alpine:3.18`
- No mutual authentication between services — any container on the Docker network can call any other
- JWT signing keys and TLS keys share the same keystore (`makani_keystore.p12`)
- The `certificate-authority` module exists as a skeleton with OpenAPI spec but no implementation

The threat model requires:
- Cryptographic service identity (not just network-level isolation)
- Protection against lateral movement if a single container is compromised
- Defense against rogue certificate issuance
- Encrypted inter-service communication with mutual authentication

## Decision

Implement an internal PKI with the following parameters:

### Cryptographic Choices

| Parameter | Value | Rationale |
|---|---|---|
| Root CA algorithm | ECDSA P-384 | 192-bit security equivalent; faster sign/verify than RSA-4096; compact signatures reduce cert size |
| Root CA validity | 10 years (3650 days) | Root rotation is a coordinated all-service event; long validity minimizes operational disruption |
| Leaf cert algorithm | RSA-3072 | 128-bit security equivalent; consistent with existing PKCS12/keytool infrastructure; avoids mixed-tooling complexity at the leaf level |
| Leaf cert validity | 365 days | Balances security (bounded compromise window) with operational simplicity for Phase 1; tighten to 90 days once automated renewal is proven |
| Keystore format | PKCS12 | Industry standard; consistent with existing `makani_keystore.p12` and `keytool` workflows |
| Crypto provider | Bouncy Castle `bcpkix-jdk18on` | JCA lacks direct CSR signing APIs; BC provides `PKCS10CertificationRequest` and `X509v3CertificateBuilder` |

### TLS Model

**mTLS (mutual TLS) enforced on all inter-service communication from Phase 1.**

Both client and server present certificates during TLS handshake. The server verifies the
client certificate is signed by the trusted root CA. This provides cryptographic proof of
service identity at the transport layer, independent of application-level authentication (JWT).

Spring Boot 4 configuration:
```properties
# Server-side mTLS
server.ssl.client-auth=need
server.ssl.trust-store=/certs/truststore.p12
server.ssl.trust-store-type=PKCS12
```

### Bootstrap Token Enrollment

To solve the mTLS chicken-and-egg problem (services need a cert to call the CA, but must
call the CA to get a cert), the CA exposes a one-time enrollment endpoint authenticated
by a pre-shared token rather than a client certificate.

**Enrollment flow:**
1. Docker Compose injects a unique `CA_BOOTSTRAP_TOKEN` as a Docker secret per service
2. Service starts without a client cert
3. Service calls `POST /ca/enroll` over one-way TLS, presenting the bootstrap token in
   an `Authorization: Bearer <token>` header alongside the CSR
4. CA validates the token against its allowlist, signs the CSR, and **invalidates the token**
   (one-time use)
5. CA returns the signed certificate + root CA certificate
6. Service builds its keystore/truststore and starts with mTLS capability
7. All subsequent inter-service communication uses mTLS
8. Certificate renewals use `POST /ca/sign-cert` (mTLS-authenticated, no token needed)

**Token properties:**
- Cryptographically random, 256-bit (Base64-encoded)
- Bound to a specific Common Name (CN) — token for `platform-core-api` cannot enroll as
  `notification-service`
- Single-use — invalidated immediately after successful enrollment
- Stored in the CA's persisted volume as a JSON manifest
- Generated by a CA init script at first startup or by an operator command

### Key Separation

JWT signing keys and TLS keys remain separate:
- **JWT keystore** (`makani_keystore.p12`): RSA-2048, used by `JwtTokenProvider` for token
  signing/verification. Rotation schedule independent of TLS.
- **TLS keystore** (CA-issued): RSA-3072, used for inter-service mTLS. Rotated via CA
  certificate renewal.
- **Rationale**: Coupling them means rotating one forces rotating both. JWT signing is an
  authentication concern; TLS is a transport concern. Different threat models, different
  rotation cadences.

### X.509v3 Extensions on Leaf Certificates

All CA-issued leaf certificates include:
- `keyUsage`: `digitalSignature`, `keyEncipherment` (critical)
- `extendedKeyUsage`: `serverAuth`, `clientAuth` (for mTLS)
- `basicConstraints`: `CA:FALSE` (critical — prevents leaf from acting as sub-CA)
- `subjectAltName`: Docker service hostname + `localhost` + `127.0.0.1`

### Network Isolation

The CA service runs on a Docker `internal: true` bridge network with no host port mapping.
Only services on the `akademia-internal` network can reach it. The `platform-core-api`
bridges both internal and external networks since it serves external traffic.

### Certificate Governance

- **CN Allowlist**: The CA maintains a whitelist of valid Common Names in configuration.
  CSRs with unrecognized CNs are rejected with 403.
- **Serial Number Tracking**: Monotonically increasing serial numbers persisted to the CA
  volume. Required for future CRL/OCSP support and audit trail.
- **Audit Logging**: Every certificate issuance logged with timestamp, CN, serial number,
  validity period, and requesting source. Feeds into `audit-service` module.

## Alternatives Considered

1. **RSA-4096 for Root CA** — Stronger key but consistent algorithm stack (all RSA).
   Not selected because ECDSA P-384 provides equivalent security (192-bit vs ~152-bit)
   with significantly smaller signatures and faster operations. The mixed-algorithm PKI
   (ECDSA root, RSA leaves) is fully valid in X.509 and avoids RSA-4096's ~12ms signing
   cost at the CA level.

2. **One-way TLS for Phase 1, mTLS later** — Simpler initial implementation relying on
   Docker network isolation as the trust boundary. Not selected because network isolation
   is a necessary but insufficient control — a compromised container on the internal
   network could impersonate any service without cryptographic identity verification.
   The bootstrap token mechanism makes mTLS-from-day-one feasible.

3. **Pre-provisioned bootstrap certificates** — CA generates short-lived client certs at
   startup, mounted as Docker secrets, used for initial mTLS enrollment. Not selected
   because it adds a dedicated init container, ephemeral cert generation logic, and
   volume coordination that the simpler bootstrap token approach avoids.

4. **Network trust for CA enrollment** — Accept unauthenticated enrollment from any
   container on the internal network. Not selected because it means any compromised
   container can request a certificate with any CN, enabling impersonation attacks
   within the service mesh.

5. **Unified JWT + TLS keystore** — Single keystore for both token signing and mTLS.
   Not selected because it couples two independent security concerns with different
   rotation schedules and threat models.

## Consequences

### Positive
- Cryptographic service identity from Phase 1 — no reliance on network isolation alone
- Bootstrap token enrollment solves mTLS chicken-and-egg cleanly
- ECDSA P-384 root CA provides 192-bit security with compact signatures
- RSA-3072 leaf certs maintain compatibility with existing PKCS12 tooling
- CN allowlisting prevents rogue certificate issuance
- Serial number tracking enables future revocation infrastructure
- Key separation allows independent rotation of JWT and TLS credentials

### Negative
- Mixed-algorithm PKI (ECDSA root + RSA leaves) adds conceptual complexity
- Bootstrap tokens are another secret to manage per service
- mTLS from Phase 1 increases initial implementation effort vs one-way TLS
- Bouncy Castle dependency required for ECDSA CA + CSR signing

### Neutral
- OpenAPI spec already defines the `POST /ca/sign-cert` and `GET /ca/ca.crt` endpoints;
  a new `POST /ca/enroll` endpoint is needed for token-based enrollment
- Docker Compose must be restructured with `internal: true` network and Docker secrets
- Existing `certs_makani.rtf` workflow becomes obsolete for inter-service TLS
  (retained only for local development JWT keystore)
